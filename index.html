Parallel Hash Tables

TITLE. Parallel Hash Tables. Brought to you by Manu Garg, Andrew Butko.

SUMMARY. We are going to implement concurrent hash tables that use coarse grained/fine grained locks. We are also going to implement lock-free hash tables. Finally we're going to analyze the implementations of each of them by testing them on different programs with different data access patterns.

BACKGROUND. A hash table is a data structure that maps keys to values and allows fast accesses to the data. More on hash tables: Hash Tables

THE CHALLENGE. The problem is challenging because lock-free data structures are hard. In general implementing a correct version of the same and avoiding race conditions will itself be a challenge. Moreover if we implement the hash tables with separate chaining then deletion is going to be an issue. Another problem would be resizing the hash table on an insert and ensuring that is still lock-free.

RESOURCES. We will be using C++ for the coding aspects of this assignment and the Gates/Latedays clusters for testing/analyzing the performance of the code. We will most likely start the code from scratch but will be using ideas from research papers to supplement the implementation. This research paper looks promising.

GOALS AND DELIVERABLES.

  • PLAN TO ACHIEVE: We plan to implement several versions of locking on hash tables (coarse, fine) along with a lock free version based on the paper above. Along with this, we plan to do a performance analysis comparing these implementations and would like to achieve similar results to the listed paper. A similar project was completed in a previous semester, so this should be achieveable.
  • HOPE TO ACHIEVE: If we find time, we'd like to investigate implementing a binary decision diagram interface on top of the parallel hash tables. We also plan to investigate using the RMC system for implementing our system.
  • Because the research paper shows a performance analysis for different workloads, we will use a similar set of tests to check our performance. We can present our results as a speedup graph and hopefully meet or exceed these results.

PLATFORM CHOICE. We will implement this using C++ (as this was the suggested comparison for RMC) and tentatively do performance testing on the Gates clusters or Latedays. We will figure out which machines best match what was used in the research paper for testing and go with that to have a better measure of our performance.
